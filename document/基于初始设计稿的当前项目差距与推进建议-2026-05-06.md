## 1. 结论摘要

结合两份初始设计稿与当前仓库实现，我对项目现状的重新判断是：

- 这个项目已经明显越过“概念验证”阶段，主体流程已经可以本地跑通，也具备基本的治理闭环。
- 初始设计稿中的主骨架已经大体落地：`Harness / Local First / MCP 事实供给 / Baseline / Feature Brief / Design Pack / Task Slice / Gate 1~5 / Release Gate / sync-baseline` 都已有真实实现，而不只是文档想法。
- 当前最主要的问题，已经不再是“有没有入口、有没有命令”，而是“事实源够不够强、组件和数据源边界收得够不够严、Gate 5 能不能真正承担业务准入、文档口径是否和代码状态同步”。
- 与 4 月底相比，项目健康度已经明显上升：`pytest -q` 已验证为 `129 passed`，fixture 矩阵已扩展到 `7` 个样例，并且已经从“正向样例集”扩到了“治理回归样例集”；`doctor.ps1` 也已经包含 Gate smoke test。
- 因此，后续推进的重点不应该再是继续堆更多流程名词，而应该集中在“可信度硬化”“组件/资源隔离收口”“真实业务验证升级”“治理文档同步”四条主线。

一句话判断：这个项目已经具备“团队试点可运行”的基础，但离“生产级可依赖治理工具”还差最后几层硬化工作。

- baseline、实现态/设计态索引、`project-console`、`flow-overview`、`project-next` 等运营视角能力已具雏形。
- `strict`、`evidence`、`hash`、`freshness`、`source_signature`、`risk_tier` 等治理字段已进入真实报告和校验逻辑。

### 3.2 已经部分落地，但 backlog 文档口径滞后的部分

这里需要特别指出：当前仓库里已经有一些能力落地了，但 `docs/sdd-architecture-open-items.md` 等跟踪文档里仍保留着旧表述，容易让判断失真。

- 高风险 `attached_execution`
  - Gate 5 / Release Gate / `validate_reports.py` 已经对高风险 feature 强化要求，不再只是可选增强。

虽然上面这些入口已经落地，但它们里仍有不少只完成了“第一层”，还没有真正形成生产级闭环。当前最需要继续推进的，就是这些“已起步但未收口”的部分。

## 4. 当前项目的主要不足

下面这些不足，是我结合两份初始设计稿、当前代码、现有样例矩阵与回归测试后，认为最影响项目继续推进质量的部分。

### 4.1 P0：事实源可信度还不够强

当前状态：

- `project-explorer` 现在已经有较强的轻量 Java 扫描能力，也能输出 `scanner`、`evidence_level`、`confidence`、`unsupported_features` 等元信息。
- 但底层主干仍然偏 lexical / lightweight 扫描，尚未升级到 Java AST / symbol 级别。
- 对 Lombok、继承方法、复杂泛型、MyBatis XML、JPA Entity、代理类等老项目高频结构，可信度仍然不够高。
- Gate 2 / Gate 5 虽然已经能对低可信事实源降级或阻断，但“知道自己不够可靠”和“本身足够可靠”仍不是一回事。

为什么这是 P0：

- 初始设计稿最核心的价值主张之一，就是不允许 AI 凭空假设工程事实。
- 如果 `module-map.json` 和实现追溯本身只是一份中等可信快照，那么后续所有 Gate 都只能做到“条件可信”，还达不到强治理。

建议工作：

- 将 `project-explorer` 从当前轻量扫描升级到 JavaParser / JDT / `javac` symbol API 级别。
- 将 `public_methods` 从“方法名级”升级到“参数签名级”。
- 补齐 MyBatis XML、JPA Entity、注解路由与真实实现之间的映射。
- 让 Gate 报告更显式地区分“因为事实不足而降级”与“因为真实不一致而失败”。

### 4.2 P0：组件维度与资源唯一键模型仍未彻底收口

当前状态：

- `components[]` 和 `affected_components` 已经不再是纯设计概念，部分流程已经支持。
- Gate 2 / Gate 5 也已经开始具备“只看受影响组件”的能力。
- 但 baseline、索引、冲突检测、schema-context 唯一键、module-map 唯一键等底层模型，还没有完整升级成稳定的 component-aware / datasource-aware 体系。
- 也就是说，组件维度入口已经有了，但底层隔离模型还没有完全跟上。

为什么这是 P0：

- 初始设计稿虽然起步于单项目视角，但目标显然不是只服务一个小样例仓库，而是支撑团队级、多模块、多服务、多库场景。
- 如果组件隔离只停留在 feature 入口和 Gate 过滤层，而没有深入 baseline 和资源主键层，规模一上来就容易相互污染。

建议工作：

- 完整收口 `components[]` attached project 模型，并明确根级配置与组件级配置的关系。
- 将 `module-map` 唯一键推进到 `component_id + fqn`。
- 将 `schema-context` 唯一键推进到 `db_type + connection_name + schema_name + table_name`，必要时加 `resource_key`。
- 升级冲突检测粒度到 `component + resource_key`，而不是继续停留在裸 `table_name` / path 级别。

### 4.3 P0：统一事实等级与 strict 语义还没有完全贯穿

当前状态：

- evidence、hash、freshness、source_signature 已经进入 baseline 和 Gate 报告。
- strict 模式也已经接入多条主流程。
- 但当前还没有做到“每一个关键结论都明确标注依赖的是哪一级事实源”，尤其是 class / table / field / test 级别的逐项证据口径还不够统一。
- `polyquery` 的 stricter 默认策略也还没有完全变成所有主推荐入口的默认行为。

为什么这是 P0：

- 初始设计稿强调的是“可解释、可追责、可审阅”，而不是“脚本跑完了所以算过了”。
- 如果 strict 的含义不够一致，团队就很难把这些报告当成真正的准入证据。

建议工作：

- 统一 L1 / L2 / L3 事实等级模型，并让 Gate 2 / Gate 5 输出逐项证据等级。
- 让高风险 feature 在 strict 路径下禁止低等级事实源直接支撑 PASS。
- 将 `polyquery-fallback fail` 风格的更严格策略贯穿主推荐命令、CI 示例和 onboarding 文档。

### 4.4 P1：Gate 5 已经明显增强，但还没有完全升级成“真实业务测试准入层”

当前状态：

- Gate 5 已经支持 `attached_execution`、真实测试覆盖扫描、实现追溯、TODO 阻断、组件范围过滤。
- 高风险 feature 已经会要求 `attached_execution_required=true`，Release Gate 和 `validate-reports` 也会消费这个约束。
- 这说明 Gate 5 已经不再只是设计验证测试骨架检查器，能力层级已经上来了。
- 但它距离“真实业务准入层”还差两步：
  - 真实测试到 REQ 的跨语言、跨框架映射规范还不够完整。
  - 非 strict 路径下的默认要求、组件级执行与更深的业务测试覆盖口径还未完全收口。

建议工作：

- 明确跨语言 `REQ-ID -> 真实测试` 标注规范。
- 继续增强 `real_test_req_coverage` 的消费逻辑，把它从报告项推进到准入项。
- 在组件模型彻底收口后，按组件执行 `verification_commands` 并输出组件级结果。

### 4.5 P1：CI / Release 的严格推荐流水线还没有完全收口

当前状态：

- 项目已经有 GitHub Actions 和报告校验能力。
- 但“团队推荐如何跑严格版 SDD 流水线”这件事，还没有形成一条统一、强约束、对外可复制的标准链路。
- 报告 hash 一致性校验也还没有完全做成默认治理规则。

建议工作：

- 明确一条标准严格链路，例如：
  - `refresh-baseline`
  - `design-gates`
  - `implementation-gates --strict`
  - `release-gate`
  - `validate-all-reports`
- 为 CI / Jenkins / GitHub Actions 提供统一示例。
- 在 `validate_reports.py` 中继续补齐 hash 一致性校验闭环。

### 4.6 P1：异常 / 豁免治理还没有形成正式模型

当前状态：

- 当前文档已经开始强调节奏适配和不要一刀切。
- 但对紧急修复、例外发布、临时绕过的记录、审批、补录要求，还缺少正式数据模型与 Gate 消费逻辑。

建议工作：

- 建立 `exception.json` 或等价的豁免记录入口。
- 明确豁免最小补录要求：原因、审批、补设计、补测试、补 baseline。
- 让 Release Gate 能识别并校验这些记录。

### 4.7 P1：fixture 矩阵已经升级，但仍需要继续从“样例集”走向“治理回归样例集”

当前状态：

- 当前 7 个 fixture 已经覆盖轻量 API、标准 DB 变更、多组件、高风险支付、polyquery snapshot、治理 FAIL、治理 WARN。
- 这比之前的样例深度已经强很多。
- 但如果要把这套项目作为长期治理工具，仍需要更多失败原因分型和更长链路样例。

建议工作：

- 继续扩展 FAIL / WARN fixture：
  - 低可信事实源 FAIL
  - stale baseline WARN / FAIL
  - schema 唯一键冲突
  - 组件隔离错误
  - MyBatis / JPA 映射失真
  - REQ 覆盖不足但 attached execution 成功
- 增加更接近真实业务项目的长链路 fixture。

### 4.8 P1：当前文档口径与代码状态存在漂移

当前状态：

- 现在最大的文档问题，不是文档少，而是有几份文档仍在用旧判断描述已经变化的实现状态。
- 例如：
  - `docs/sdd-architecture-open-items.md` 仍写着 `affected_components` 尚未支持，但代码和测试已经支持。
  - 同文档仍写着 doctor 尚未包含 Gate smoke test，但实际脚本已经包含。
- 这类漂移会直接影响后续优先级判断。

建议工作：

- 将初始设计稿明确定位为“历史设计意图文档”。
- 将 `docs/sdd-architecture-open-items.md` 维护为唯一主 open items 清单，但要求与代码状态定期同步。
- 将本文档作为阶段性复盘与优先级重排文档，而不是长期 backlog 主清单。

## 5. 如果继续推进，建议按什么顺序做

### 第一阶段：先把“可信度主链路”补硬

优先级：最高

建议聚焦：

- AST / symbol 级 `project-explorer`
- MyBatis / JPA / 参数签名级实现映射
- 统一证据等级模型
- strict 语义和 stricter polyquery 默认策略收口

完成标志：

- 高风险 feature 的 PASS / FAIL 可以明确说明依赖了什么级别的事实源。
- Gate 2 / Gate 5 的“可解释性”和“可审计性”明显提升。

### 第二阶段：把组件维度和资源唯一键模型收口

优先级：高

建议聚焦：

- component-aware baseline
- component-aware schema-context
- module-map / schema-context 唯一键升级
- 组件粒度冲突检测

完成标志：

- 多组件、多服务、多数据源项目可以在同一套 SDD 流程下运行，且事实源互不污染。

### 第三阶段：把 Gate 5 真正推到业务准入层

优先级：高

建议聚焦：

- `REQ-ID -> 真实测试` 标准化映射
- 真实测试覆盖结果从“报告字段”升级到“准入字段”
- 组件级 `verification_commands`
- 高风险默认治理口径进一步统一

完成标志：

- 对高风险 feature，团队可以把 Gate 5 + Release Gate 当成真正的上线前治理门禁，而不是辅助参考报告。

### 第四阶段：收口 CI、豁免治理和回归样例矩阵

优先级：中高

建议聚焦：

- 标准 strict CI 流水线
- hash 校验闭环
- exception / waiver 模型
- 更完整的 FAIL / WARN fixture 族群

完成标志：

- 项目进入“长期可维护治理工具”的状态，而不是仍然依赖作者脑内口径。

## 7. 最终判断

这个项目当前最大的优点，不是功能点多，而是它已经把最难的一段骨架搭出来了：

- 有本地优先主流程
- 有事实源
- 有 baseline
- 有设计与实现态产物
- 有 Gate
- 有 attached project
- 有样例矩阵
- 有自动化测试

它当前最大的不足，也不再是“还没开始做治理”，而是距离“生产级可信治理”还差最后几层硬化：

- 事实源要更强
- 组件与资源隔离要更深
- Gate 5 要更像真实业务准入层
- 文档与 backlog 口径要和代码状态同步

因此，如果继续推进，我建议把资源优先投到这四件事上：

1. AST / symbol 级 `project-explorer` 与 MyBatis / JPA / 参数签名级映射。
2. component-aware baseline 与资源唯一键模型收口。
3. 高风险 strict / Gate 5 / 真实测试覆盖的准入化。
4. CI、豁免治理、治理回归 fixture 与主文档口径同步。

做完这四层之后，这个项目才会真正从“团队试点可跑”进入“团队可以长期依赖”的阶段。
