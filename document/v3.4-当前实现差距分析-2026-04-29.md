## 3. 本轮已补齐的差距

## 3.1 Brownfield Gate 2 真实性校验

### 原差距

Brownfield 下 Gate 2 之前主要检查：

- `feature-brief.md` 中 REQ-ID 是否存在
- `design-pack/` 是否存在
- `design-pack/` 是否覆盖 REQ-ID
- Markdown 设计包是否过薄

还没有深入消费 `module-map.json`、`schema-context.json`、`sdd-index-real.json` 等事实源。

### 当前状态

已在 `scripts/check_design_truthfulness.py` 中增强 Brownfield 校验：

- 从 `design-vN.md` 提取 Mermaid participant / classDiagram 类名。
- 校验设计参与类是否存在于 `module-map.json`。
- 从 `接口契约.openapi.yaml` 提取 API path。
- 校验 API path 是否与实现态 baseline 中其他 feature 冲突。
- 对已实现的同 feature 设计，允许重跑门禁，避免历史实现态记录误判为自冲突。
- 从 `数据模型.md` 提取表名。
- 结合 `schema-context.json` 与 `数据库变更.sql` 判断表名是否可追溯或明确新增。

### 当前结论

此项已从 P0 缺口降级为 P2 深化项。

后续可继续增强：

- 方法级存在性校验。
- 字段级存在性校验。
- 更严格的新表 / 旧表 / 字段新增标记规则。

## 3.7 polyquery 实时数据库事实源接入

### 原差距

v3.4 中 `polyquery` 应作为数据库实时元数据事实源，为 `schema-context.json`、Gate 2 和 `sdd-generation` 提供表、字段、索引等信息。

之前 `schema-context.json` 主要来自：

- 附着项目 SQL / 资源目录
- 设计包中的 `数据模型.md`
- 设计包中的 `数据库变更.sql`

因此它更接近本地快照，不是真正的实时数据库事实。

### 当前状态

已完成项目侧接入：

- 新增 `scripts/polyquery_adapter.py`
- 新增 `config/polyquery.example.json`
- 新增 `docs/polyquery-integration.md`
- `scripts/refresh_schema_context.py` 支持 polyquery 模式
- `scripts/run_pipeline.py refresh-schema-context` 已透出 polyquery 参数
- 保留 polyquery 原版多数据库、多数据源配置：同一个 `*_CONFIGS` 可配置多个 `connection_name`
- `sources` 已降级为高级可选项；默认可通过 `--auto-discover specs/<feature>` 交给 SDD/agent 自动发现需要沉淀的表

当前支持：

- `--from-polyquery`
- `--polyquery-config`
- `--polyquery-snapshot`
- `--auto-discover`
- `--polyquery-fallback local|fail`

实时刷新命令：

```powershell
python scripts\run_pipeline.py refresh-schema-context --from-polyquery --polyquery-config config\polyquery.json --polyquery-fallback fail
```

自动发现刷新命令：

```powershell
python scripts\run_pipeline.py refresh-schema-context --from-polyquery --auto-discover specs\your-feature --polyquery-config config\polyquery.json --polyquery-fallback fail
```

离线 / CI snapshot 命令：

```powershell
python scripts\run_pipeline.py refresh-schema-context --from-polyquery --polyquery-snapshot .spec\polyquery.snapshot.json
```

### 当前结论

此项已完成项目侧最小闭环。

剩余工作不再是“项目没有接入 polyquery”，而是团队接入配置：

- 为各业务项目准备 `config/polyquery.json` 或等价 MCP 环境配置。
- 确认只读账号、开发/测试库、多数据源命名。
- 仅在自动发现无法满足时，补充 `sources` 表清单或匹配规则。
- 在团队流程中约定 Gate 2 前优先执行 polyquery schema refresh。

## 4. 仍然欠缺或需要继续深化的部分

## 4.1 Design Pack YAML / SQL / OpenAPI lint 已增强，仍可继续深化

### 当前实现

当前已有：

- `.spec/schemas/reports/*`
- `.spec/schemas/design-pack/*`
- `scripts/sdd_yaml.py`
- `scripts/check_design_pack.py`
- OpenAPI operation 级检查
- SQL UP / DOWN 回滚动作检查

已增强的确定性规则：

- OpenAPI 每个 operation 必须有 `operationId`。
- OpenAPI 每个 operation 必须有 `responses` 和至少一个 HTTP 状态码。
- OpenAPI `operationId` 不允许重复。
- OpenAPI 路径参数 `{param}` 必须声明为 `in: path`。
- SQL `-- UP` 后必须有实际迁移语句。
- SQL `-- DOWN` / `-- ROLLBACK` 后必须有实际回滚语句。
- `CREATE TABLE` 必须有对应 `DROP TABLE` 回滚。
- `ALTER TABLE ... ADD COLUMN` 必须有对应 `DROP COLUMN` 回滚。
- `feature-brief.md` / `release-plan.md` 关键字段已改用共享 YAML 解析适配层读取。
- `sdd_yaml.py` 在存在 PyYAML 时优先使用 PyYAML；未安装 PyYAML 时使用项目内轻量解析器，覆盖当前 feature brief / release plan 的嵌套 map、list、quoted scalar、boolean 等结构。

仍然偏轻量的部分：

- 当前环境未安装 PyYAML，因此仍依赖项目内轻量 YAML fallback。
- OpenAPI 还不是完整 OpenAPI 3.x 标准校验。
- SQL 还不是数据库方言级语法解析。

### 剩余差距

已经完成关键 YAML 字段的共享解析收口，但还不是完整规范级 lint。

建议逐步接入：

- OpenAPI 标准校验。
- SQL 语法 / 回滚脚本结构校验。
- 在依赖允许时引入 PyYAML / jsonschema，对 feature brief、release plan、design-pack 做 schema 级严格校验。

## 4.2 Gate 5 真实实现验证依赖项目配置

### 当前实现

Gate 5 已支持 attached project 的 `verification_commands`：

```json
{
  "verification_commands": [
    {
      "name": "unit-test",
      "command": ["mvn", "test"]
    }
  ]
}
```

任一命令失败，Gate 5 会判定为 `FAIL`。

同时已支持强制 attached project 验证：

```powershell
python scripts\run_pipeline.py gate5 specs\your-feature --require-attached-execution
```

启用该参数后，如果未配置 attached project 或未配置 `verification_commands`，Gate 5 会直接 `FAIL`。

### 剩余差距

这项能力依赖各业务项目显式配置。

如果未启用 `--require-attached-execution`，Gate 5 仍主要依赖：

- SDD 生成的设计验证骨架
- TODO 消除检查
- Java / Python 测试执行
- `module-map.json` 实现追溯

建议在团队接入规范中要求每个业务项目配置至少一个真实测试命令，并在试点/CI 中启用 `--require-attached-execution`。
