## 1. 拆分目标

这份文档不是再做一轮架构分析，而是把上一份“差距与建议”继续拆成可以直接排期、拆 issue、分配负责人的执行清单。

拆分原则：

- 先补可信度和边界，再补规模化和体验。
- 先做高复用地基，再做依赖这些地基的增强项。
- 每个工作包都必须有明确交付物、依赖关系和验收标准。
- 避免把“继续写文档”误当成“能力已经落地”。

---

## 2. 推荐推进节奏

建议按四个里程碑推进：

| 里程碑 | 目标 | 结果判断 |
| --- | --- | --- |
| M1 | 可信度主链路补硬 | 高风险场景的 Gate 2 / Gate 5 结论更可信 |
| M2 | 多组件与多数据源建模 | 多服务项目可在同一套 SDD 工具链下运行 |
| M3 | Gate 5 真实业务准入化 | 高风险 feature 可把 Gate 5 作为真实业务测试门禁 |
| M4 | 文档、样例、发布与运营收口 | 新成员可快速接入，团队可长期维护 |

建议优先顺序：

1. `P0-WP1` AST 级事实源增强
2. `P0-WP2` strict / evidence / polyquery 策略贯穿
3. `P0-WP3` component-aware baseline
4. `P1-WP1` Gate 5 真实业务测试准入
5. `P1-WP2` 双模式、豁免机制与样例矩阵
6. `P1-WP3` 字段/方法/契约级校验收口
7. `P1-WP4` 文档主口径收口
8. `P2-WP1` 产品化分发与运营看板

## 3. 工作包拆分

## 3.1 P0-WP1：AST 级事实源增强

目标：

- 把 `project-explorer` 从轻量 lexical 扫描升级到更稳定的 AST / 符号级扫描。

依赖：

- 无，是后续多个工作包的地基。

主要交付物：

- `project-explorer` 新扫描器版本
- 升级后的 `module-map.json` 结构
- 新增/补充回归测试

任务拆分：

1. 设计新的扫描结果模型，明确 `scanner`、`confidence`、`unsupported_features`、`public_methods`、`fqn`、继承方法等字段。
2. 在 `mcp-servers/project-explorer` 中接入 JavaParser / JDT / `javac` symbol API 其中一种可控方案，先完成最小可用实现。
3. 补同名类、内部类、record、interface、enum、继承方法、泛型方法的样例和测试。
4. 增加 Lombok、MyBatis XML、JPA Entity、代理类等“低可信特征”的识别和显式标记。
5. 让 `refresh_module_map.py` 和 Gate 消费端兼容新旧结构。

验收标准：

- `module-map.json` 能稳定区分同名类 FQN。
- `public_methods` 不再只停留在方法名级，至少支持更可靠的方法签名抽取。
- Gate 2 / Gate 5 能根据 `confidence` 做更清晰的 WARN / FAIL 决策。
- 对至少一个复杂 Java fixture 验证扫描结果优于当前 lexical 方案。

- [server.ts](</D:/project/SDD/mcp-servers/project-explorer/src/server.ts:1>)
- [refresh_module_map.py](</D:/project/SDD/scripts/refresh_module_map.py:1>)

## 3.2 P0-WP2：strict / evidence / polyquery 策略贯穿

目标：

- 让“事实等级”“strict 模式”“polyquery fallback 策略”真正贯穿主流程，而不是分散在局部脚本里。

依赖：

- 与 `P0-WP1` 并行可做，但完成度会受 `module-map` 可信度提升影响。

主要交付物：

- 统一 evidence 等级口径
- strict 推荐命令
- 主流程中的严格 polyquery 策略

任务拆分：

1. 定义并落地 `L1/L2/L3` 事实等级模型，至少覆盖 design、design-pack、module-map、schema-context、verify-report。
2. 统一 Gate 2 / Gate 5 报告结构，使关键结论都能附上 evidence source。
3. 检查 `design-gates`、`implementation-gates`、`full-flow`、CI 推荐路径，补 strict 模式透传。
4. 对高风险 feature 统一约束：`local-fallback` 不能作为默认可接受路径。
5. 在文档和命令帮助里明确 strict 与非 strict 的边界。

验收标准：

- 高风险 feature 的 Gate 报告能看出每个核心结论来自哪一级事实。
- `refresh-schema-context --from-polyquery --polyquery-fallback fail` 能进入推荐主链路。
- strict 模式下，低可信或 fallback 事实不会被误判成可直接 PASS。

## 3.3 P0-WP3：component-aware baseline

目标：

- 把当前单 attached project 模型升级为支持多组件、多服务、多数据源的 baseline 模型。

依赖：

- 建议排在 `P0-WP1` 和 `P0-WP2` 之后，这样可以直接基于更强的事实源和 strict 策略建模。

主要交付物：

- `components[]` attached project 配置模型
- component-aware baseline 目录与唯一键策略
- 组件粒度 Gate 检查和报告

任务拆分：

1. 设计 `attached-project.json` 的兼容扩展方案，同时兼容现有单项目配置。
2. 为 `feature-brief.md` 或 design 产物增加 `affected_components` 入口。
3. 升级 `baseline_paths.py`、`refresh_module_map.py`、`refresh_schema_context.py`，支持按 component 分区写入。
4. 升级 `schema-context` 和 `module-map` 的唯一键模型，避免跨 component / datasource / schema 冲突。
5. 升级 Gate 2 / Gate 5 / project-console，使结果能按受影响组件展示。

验收标准：

- 一个包含两个以上组件的 attached project 可以生成隔离的 baseline 事实。
- 冲突报告可以定位到具体 component / datasource / schema。
- `verification_commands` 具备后续按 component 执行的结构前提。

## 3.4 P1-WP1：Gate 5 真实业务测试准入化

目标：

- 让 Gate 5 从“设计验证测试检查器”进一步变成“真实业务测试准入层”。

依赖：

- 依赖 `P0-WP2` 的 strict / evidence 规则。
- 最好在 `P0-WP3` 完成后再做 component 级细化。

主要交付物：

- 高风险 feature 的默认 `verification_commands` 要求
- 更强的 `REQ-ID -> 真实测试` 追溯
- release gate 对真实测试结果的消费

任务拆分：

1. 重新定义高风险 feature 在非 strict / strict 模式下对 `verification_commands` 的要求。
2. 扩展 Gate 5 报告，使 attached execution 不只是命令结果，还能和需求覆盖或组件关联起来。
3. 让 release gate 明确消费 attached execution 的结果，不再只做被动展示。
4. 在多组件模型完成后，支持按 component 分别运行验证命令。
5. 补高风险 feature 的完整样例和回归测试。

验收标准：

- 高风险 feature 未配置真实验证命令时，不能被误判成“已具备上线级验证”。
- release gate 对真实业务测试失败能明确阻断。
- 团队能区分“设计测试通过”和“业务测试通过”。

建议影响文件：

- [check_design_test_coverage.py](</D:/project/SDD/scripts/check_design_test_coverage.py:1>)
- [release_gate.py](</D:/project/SDD/scripts/release_gate.py:1>)

## 3.5 P1-WP2：双模式、豁免机制与样例矩阵

- 收口 Greenfield / Brownfield 双模式的治理深度差异，并补齐团队真实使用时的节奏适配。

- 紧急修复路径有记录，有补录要求，有 release 校验。

## 3.6 P1-WP3：字段/方法/契约级校验收口

目标：

- 把已经起步的字段级、方法级、契约级校验补到“更像生产级门禁”的程度。

依赖：

- 强依赖 `P0-WP1` 的 AST/签名增强。

主要交付物：

- 更强的字段校验
- MyBatis / JPA 映射
- OpenAPI 与 Controller 更稳定的一致性校验

任务拆分：

1. 建立 MyBatis XML / JPA Entity 与数据模型的最小映射方案。
2. 升级方法校验到参数类型级签名匹配。
3. 强化 OpenAPI path/method/operationId 与 Controller 的一致性检查。
4. 为这些规则补清晰的报错与测试样例。

验收标准：

- 设计中引用不存在字段时，Gate 2 明确失败且能定位字段。
- 设计中引用方法签名不匹配时，Gate 5 至少能精确 WARN 或 FAIL。
- OpenAPI 与实现不一致时，报告能定位到具体 path / method / operationId。

## 4. 依赖关系图

推荐依赖顺序：

```text
P0-WP1 AST事实源增强
  -> P0-WP2 strict/evidence贯穿
  -> P1-WP3 字段/方法/契约级校验收口

P0-WP2 strict/evidence贯穿
  -> P1-WP1 Gate5真实业务测试准入化

P0-WP3 component-aware baseline
  -> P1-WP1 Gate5按组件执行验证

P1-WP2 双模式/豁免/样例矩阵
  -> P1-WP4 文档主口径收口

以上工作包基本稳定后
  -> P2-WP1 产品化发布与运营看板
```

不建议过早投入的事项：

- 继续扩充新命令但不补事实源可信度。
- 过早做“SDD -> Code 自动生成”类高不确定性扩展。
- 在 component 模型未落地前，过度建设复杂多服务样例。

## 6. 最终建议

如果团队接下来只能投入一到两个迭代，我建议目标不要写成“继续完善 SDD”，而写成更具体的三件事：

1. 把事实源做强，让 Gate 结论更可信。
2. 把多组件模型补齐，让真实项目能接进来。
3. 把高风险 Gate 5 变成真正的业务测试门禁。

只要这三件事推进到位，这个项目就会从“试点可跑工具”明显跃迁到“团队可以依赖的工程治理底座”。
