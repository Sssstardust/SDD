# AI 驱动规范开发主规范模板

**版本:** `vX.Y`  
**发布日期:** `YYYY-MM-DD`  
**适用范围:** `项目组 / 事业部 / 平台团队`  
**维护人:** `姓名 / 角色`  
**状态:** `Draft | Review | Approved`

---

## 1. 文档目标

本规范用于定义本团队在软件开发过程中如何使用 AI 辅助完成：

- 需求澄清
- 技术设计
- 任务拆分
- 验证闭环
- 规范审计

本规范的目标不是替代开发流程，而是为 AI 参与开发提供统一的输入、产物、门禁和追溯机制。

---

## 2. 适用场景

### 2.1 支持的项目模式

- `brownfield`：存量项目，在已有代码和数据库基础上迭代开发
- `greenfield`：新建项目，从架构和脚手架开始构建

### 2.2 必须使用本规范的场景

- 新增对外接口
- 涉及数据库表结构变更
- 涉及跨服务调用
- 涉及消息队列、缓存、支付、幂等等高风险能力
- 涉及核心业务规则调整

### 2.3 可选使用本规范的场景

- 一般业务优化
- 中小型功能改造
- 内部工具类功能

### 2.4 可豁免场景

- 纯文案调整
- 不涉及行为变化的重构
- 小型 bugfix

---

## 3. 核心原则

### 3.1 Baseline First

AI 生成内容必须优先基于真实工程上下文或规划型基线，不允许凭空假设。

### 3.2 Capability Driven

模板、产物、门禁由 `capability_tags` 驱动，而不是只靠功能名称或单一类型字段。

### 3.3 Reviewable by Humans

所有关键产物都必须能被人工审阅、讨论、签字和追责。

### 3.4 Traceable End-to-End

需求、设计、任务、测试、实现之间必须能够建立追溯链。

### 3.5 Local First

关键验证优先在本地触发，CI 只做复核，不作为唯一事实来源。

## 4. 角色与职责

| 角色 | 核心职责 |
| --- | --- |
| 架构负责人 | 批准主规范、审批高风险设计、解释架构红线 |
| 平台/工具负责人 | 维护脚本、模板、Gate、Baseline |
| 业务开发 | 产出 feature brief、设计文档、切片任务并完成实现 |
| 测试负责人 | 审阅测试骨架、验证覆盖规则、维护验收标准 |
| AI Agent | 生成草稿、组装上下文、执行规则校验、输出结构化产物 |

---

## 5. 工作流总览

```text
Feature Brief
  -> Draft Design
  -> Gate 1 / Gate 2 / Gate 3
  -> 风险审批（仅 High）
  -> 写入设计态索引
  -> Task Slice
  -> Gate 4 测试骨架生成
  -> code 完成
  -> Gate 5 覆盖验证
  -> Baseline 同步
```

## 6. 关键输入与统一字段

### 6.1 feature-brief 必填字段

```yaml
project_mode: brownfield
project_mode_source: agent
project_mode_confidence: 0.85
project_mode_confirmed_by: zhangsan

feature_name: order-create
feature_type: sync
one_liner: "一句话描述功能目标"

capability_tags:
  - api
  - db-change

risk_tier: high

requirements:
  - req_id: REQ-001
    priority: P0
    description: "需求描述"
```

### 6.2 统一枚举

- `project_mode`: `brownfield | greenfield | hybrid`
- `risk_tier`: `low | high`
- `priority`: `P0 | P1 | P2`

## 7. Baseline 规范

### 7.1 Baseline 目录

```text
.spec/baseline/
├── sdd-index-design.json
├── sdd-index-real.json
├── module-map.json
├── schema-context.json
├── constitution.md
└── tech-debt.md
```

### 7.2 设计态索引状态

- `ACTIVE`
- `SUPERSEDED`
- `CANCELLED`
- `IMPLEMENTED`

### 7.3 基本规则

- 只有 `ACTIVE` 参与占位校验
- 高风险设计必须审批通过后才能进入设计态索引
- Gate 5 通过后，设计态条目才能迁移到实现态

## 9. Task Slice 规范

### 9.1 切片分类

- 垂直切片：面向业务路径
- 横切切片：面向基础设施、监控、配置、权限等横切事项

### 9.2 最小结构

```yaml
slice_id: SLICE-001-BIZ
depends_on: []
req_ids:
  - REQ-001

  - "可测量的验收条件"
test_spec:
  type: integration
  framework: junit5
```

## 10. Gate 规范

### Gate 1：结构完整性

检查：

- 必填章节
- 必填产物
- 模板最小结构

### Gate 2：真实性 / 结构约束

- `brownfield`：校验类、字段、接口、表是否真实存在
- `greenfield`：校验模块边界、命名、包结构、脚手架是否符合规划

### Gate 3：架构语义审计

检查：

- 分层
- 事务边界
- 异常处理
- 依赖方向

### Gate 4：测试骨架生成

基于：

- 设计文档
- Task Slice
- REQ-ID

生成：

- 测试骨架
- REQ-ID 到测试方法映射

### Gate 5：覆盖验证

规则：

- 主触发源：本地 `python scripts/run_pipeline.py gate5 <feature>`
- CI 只做复核
- P0 未覆盖或验证失败时，阻断提交/合并

## 11. 报告目录规范

```text
reports/
└── v{N}/
    ├── approval.json
    ├── gate-report.json
    ├── gate4-skeleton.json
    └── verify-report.json
```

规则：

- 每个设计版本绑定一个报告目录
- 历史报告只归档，不覆盖

---

## 12. 本地与 CI 规则

### 12.1 本地规则

- Feature Brief 必须先过前置检查
- Gate 5 必须在 code 完成时本地执行

### 12.2 CI 规则

- 复核 Gate 5 结果
- 若缺少 `reports/v{N}/verify-report.json`，直接报错
