# design-pack 模板清单

## 1. 目的

这份文档用于统一 `design-pack/` 下人工审阅类文档的最小模板结构，约束 agent 输出，减少不同需求之间的文档风格漂移。

适用范围：

- `接口文档.md`
- `数据模型.md`
- `幂等策略.md`
- `支付状态机.md`
- `对账策略.md`
- `外部调用策略.md`

不适用范围：

- `接口契约.openapi.yaml`
- `数据库变更.sql`
- `异步事件契约.yaml`

上述文件属于机器优先产物，应分别遵循 OpenAPI、SQL、事件契约 schema，不使用 Markdown 模板。

---

## 2. 模板目录位置

```text
.spec/templates/design-pack/
├── 接口文档.template.md
├── 数据模型.template.md
├── 幂等策略.template.md
├── 支付状态机.template.md
├── 对账策略.template.md
└── 外部调用策略.template.md
```

建议规则：

- 模板文件只定义章节骨架和最小信息集
- 模板中禁止写死具体业务内容
- agent 输出时必须保留所有必填章节
- 若某章节确实不适用，需写明“不适用原因”，而不是直接删除

---

## 3. 通用写作约束

- 所有文档必须在开头声明 `feature_name`
- 所有文档必须标注适用的 `capability_tags`
- 所有文档必须引用对应的 `REQ-ID`
- 所有结论性语句都应可追溯到 `feature-brief.md` 或 `design-v{N}.md`
- 所有时间、阈值、重试次数、TTL 等配置项，必须明确写值或写明“由配置中心控制”

建议统一头部元数据：

```yaml
feature_name: order-create
design_version: design-v1.md
capability_tags:
  - api
  - db-change
req_ids:
  - REQ-001
  - REQ-002
status: Draft
owner: zhangsan
reviewer: lisi
```

## 5. 使用建议

- 先由 agent 按模板生成初稿
- 再由人工补充业务细节和待确认项
- 最终 Gate 1 负责校验结构完整性
- Gate 3 负责校验关键策略类文档的语义一致性
